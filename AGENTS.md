# AGENTS.md

本文件定义 Codex 和其他代码代理在本项目中的仓库级工作规则。

## 项目定位

- 本项目是一个科研代码仓库。
- 核心目标是在保证实验可复现、可审查、可追踪的前提下，快速迭代研究 idea。
- 优先进行小范围、聚焦、可审查、可回滚的修改。
- 除非用户明确要求且风险合理，否则不要进行大范围重构。

## 语言与沟通

- 默认使用简体中文与用户交流。
- 代码标识符、文件名、命令行参数、配置键和日志字段保持英文。
- 新增或修改的注释、docstring、脚本说明优先使用简体中文。
- 最终回复用户时，应总结：修改了哪些文件、如何运行、做了哪些验证、还存在哪些风险。

## 科研工作流

- 在为某个研究 idea 修改代码前，应先阅读 `docs/CODEMAP.md` 和 `docs/IDEAS.md` 中对应的 idea 条目。
- 新 idea 应优先通过新模块、新配置文件、实验分支或显式配置开关实现。
- 不要覆盖 baseline 行为。除非用户明确要求，否则应保持已有 baseline 可复现。
- 每次实现或测试 idea 后，应更新 `docs/EXPERIMENTS.md`，长实验都通过 nohup 挂到后台运行。
- 如果实验结果影响后续研究方向，应更新 `docs/DECISIONS.md`。
- 不要编造实验结果。没有实际运行的实验，结果必须写为 `Pending`。
- 除非用户明确要求，不要运行长时间训练任务。

## 文档地图

- `docs/IDEAS.md`：记录计划中的研究 idea。
- `docs/EXPERIMENTS.md`：记录实验、命令、指标、观察和结论。
- `docs/CODEMAP.md`：描述仓库结构和安全修改点。
- `docs/DECISIONS.md`：记录重要研究决策。
- `docs/PROMPTS.md`：保存可复用的 Codex 提示词。
- 实验相关文档统一放在 `docs/` 目录下。

## 代码修改规则

- 遵循仓库现有代码风格、目录结构和命名习惯。
- 修改范围应严格聚焦用户请求。
- 不要随意重构核心训练、评估、数据读取、模型或配置代码。
- 除非用户明确要求，不要修改、移动、删除或覆盖 raw data、cached data、pretrained checkpoints、previous results、logs、model weights 或 intermediate artifacts。
- 不要删除或覆盖实验日志、结果文件或 checkpoint。
- 涉及训练、评估、数据增强或净化流程时，应通过明确随机种子、参数来源和输出路径来保持可复现性。
- 修改 shell 脚本时，应注意变量引用、带空格路径、失败退出行为和稳定的日志文件名。

## 注释风格

- 关键函数应有注释或 docstring，说明功能、用法和实现原理。
- 避免逐行解释式注释，不要重复代码本身已经表达清楚的内容。
- 主要在复杂逻辑、实验参数、张量 shape、数据流、边界条件、随机性控制和非显然实现处添加注释。
- 如果已有英文注释与当前改动直接相关，可以改为简体中文；不要为了统一语言而重写无关注释。
- 注释应说明“为什么这样做”或“这里需要注意什么”。

## 验证

- 修改代码后，优先运行与改动直接相关的最小验证命令。
- 适用时，在 `torch` conda 环境中进行验证。
- 如果完整训练或评估成本较高，优先做语法、参数解析、导入行为或 dry-run 检查。
- 如果无法运行验证，应说明原因并指出剩余风险。

## 汇报

- 完成后简要汇报：
  - 修改了哪些文件
  - 修改目的
  - 如何运行或继续
  - 已执行的验证
  - 剩余风险或未知项
- 如果工作区已有用户改动，不要回退这些改动。只有当它们影响当前任务时才需要说明。
